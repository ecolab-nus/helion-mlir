# Non-Torch Device IR Ops Seen in the Examples

This note documents the Helion-specific FX / Device IR operators that show up in the example kernels in this repo.

Important scope note:

- These names appear in Helion's internal traced FX / Device IR, not in the final lowered MLIR in `examples/outputs/*.mlir`.
- The final MLIR has already lowered most of these ops into `linalg`, `tensor`, `scf`, `arith`, and `loom` operations.
- This document focuses on Helion-defined ops and synthetic tracing ops. Plain Python FX targets such as `operator.add`, `operator.eq`, and `operator.getitem` are not Helion-specific and are only mentioned briefly.

The definitions below were checked against the installed Helion source in:

- `.venv/lib/python3.11/site-packages/helion/language/*.py`
- `.venv/lib/python3.11/site-packages/helion/_compiler/device_ir.py`

## Where These Ops Sit in the Pipeline

For the examples in this repo, the rough lowering pipeline is:

1. User kernel Python code
2. Helion host AST walking
3. Helion FX / Device IR
4. `helion-mlir` lowering in this repo
5. Final MLIR in `examples/outputs/*.mlir`

The ops documented here belong to step 3.

## Ops Actually Seen in the Shipped Examples

From the current example graphs:

- `examples/scripts/matmul.py`:
  `_get_symnode`, `_host_tensor`, `_new_var`, `_phi`, `_for_loop`, `full`, `load`, `store`
- `examples/scripts/matmul_split_k.py`:
  `_get_symnode`, `_host_tensor`, `_new_var`, `_if`, `atomic_add`, `load`, `tile_begin`
- `examples/scripts/mamba_chunk_scan.py`:
  `_get_symnode`, `_host_tensor`, `_new_var`, `_phi`, `_for_loop`, `full`, `load`, `store`, `dot`, `tile_begin`, `tile_id`, `tile_index`, `subscript`
- `examples/scripts/attn.py`:
  `_get_symnode`, `_host_tensor`, `_new_var`, `_phi`, `_for_loop`, `_mask_to`, `full`, `load`, `store`, `subscript`

Two small clarifications:

- `hl.zeros(...)` does not appear as its own FX op. It is front-end sugar for `hl.full(..., 0, ...)`.
- Property syntax like `tile.begin`, `tile.id`, and `tile.index` becomes `tile_begin(...)`, `tile_id(...)`, and `tile_index(...)` in Device IR.

## Synthetic Tracing / Control-Flow Ops

These are internal helper ops defined in `helion.language._tracing_ops`. They are generated by Helion while tracing; users do not normally call them directly.

### `_get_symnode(debug_name: str) -> int`

Represents a symbolic scalar that needs an FX origin node.

What it means:

- FX requires symbolic values such as `torch.SymInt`, `torch.SymFloat`, and `torch.SymBool` to come from some node.
- Helion inserts `_get_symnode` lazily when it needs to materialize such a symbolic value in the graph.
- In practice this is how block sizes, symbolic dimensions, and other symbolic expressions enter the Device IR.

Parameters:

- `debug_name: str`
  Human-readable name used for debugging and node naming.
  The real semantic value is not encoded in this string.
  The actual symbolic value is stored in `node.meta["val"]`.

Notes:

- If the symbol came from a registered block size, Helion can resolve it to a `BlockSizeOrigin`.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- If the symbol has `BlockSizeOrigin` and the block size is static, lower to `arith.constant ... : index`.
- If the symbol has `BlockSizeOrigin` and the block size is dynamic, reuse the pre-emitted block-size SSA from the `loom.sym` setup done before graph walking.
- If the symbol is otherwise concretized in Helion's `shape_env`, lower to `arith.constant ... : index`.
- If none of those cases apply, the lowering currently errors out instead of emitting a generic symbolic MLIR op.

### `_host_tensor(debug_name: str) -> torch.Tensor`

Represents a tensor that originates on the host side and must be passed into the kernel as an argument.

What it means:

- This is the source node for kernel arguments and other host-origin tensors captured by tracing.
- It is also used for scalar host tensors when a scalar value is coming from a host tensor rather than from a literal.

Parameters:

- `debug_name: str`
  Host-side origin string, typically derived from `Origin.host_str()`.
  This is descriptive only.
  The actual tensor object is stored in `node.meta["val"]`.

Notes:

- Helion validates that scalar `_host_tensor` nodes only feed ops that explicitly allow host tensors.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- No dedicated MLIR op is emitted.
- The node is resolved directly to a pre-registered function argument SSA from `ctx.host_tensors`.
- For this repo, those host tensors are represented as `memref` function parameters.

### `_new_var(value) -> same type as value`

Creates a fresh SSA-style variable name for an existing value.

What it means:

- Helion inserts `_new_var` to avoid accidental aliasing when the same logical value gets renamed across loop and branch boundaries.
- Its main purpose is to make later `_phi` handling correct.

Parameters:

- `value`
  Supported by Helion for:
  `torch.Tensor`, `torch.SymInt`, `int`, `float`, `bool`, and `None`.

Notes:

- This does not change the mathematical value.
- It is purely a "fresh name / fresh identity" node in the traced IR.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- No dedicated MLIR op is emitted.
- The lowering simply forwards the operand SSA to the `_new_var` result.

### `_for_loop(graph_id: int, begin: list[int], end: list[int], args: list[object]) -> list[object]`

Represents a `for` loop, because plain FX graphs do not natively model structured control flow.

What it means:

- The loop body lives in a separate subgraph.
- The outer graph contains a single `_for_loop` node that references that body subgraph.
- Loop-carried tensor values are passed in and returned explicitly.

Parameters:

- `graph_id: int`
  Index into `HostFunction.current().device_ir.graphs`.
  This points at the traced loop-body graph, usually a `ForLoopGraphInfo`.
- `begin: list[int]`
  Per-dimension loop start.
  Even a 1-D loop is represented as a one-element list.
- `end: list[int]`
  Per-dimension loop end.
- `args: list[object]`
  Lifted outer-scope values used by the loop body.
  In practice this is mainly the tensor state that becomes loop-carried.
  Plain scalars such as `int`, `bool`, `None`, `torch.SymInt`, and `Tile` are not usually lifted here.

Result:

- Returns the loop outputs corresponding to the lifted loop-carried values.

Notes:

- The loop's tiled dimensions are not encoded directly in the op operands.
  They live in the associated `ForLoopGraphInfo.block_ids`.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- Lowered to `scf.for`, not to a custom loop op.
- The trip count is derived primarily as `arith.ceildivui(upper_bound, block_size)`.
- The lowering emits explicit `%c0` and `%c1` index constants for loop bounds and step.
- The induction variable is named `%iv_block_<block_id>`.
- Only values proven to participate in the `_phi(acc, getitem(_for_loop, i))` pattern become `iter_args`.
- The body is lowered by recursively visiting the referenced `ForLoopGraphInfo`.
- The loop terminates with `scf.yield`.
- Multi-result loops use MLIR's multiple-result form, for example `%res:2 = scf.for ...`.

### `_if(test: object, graph_id: int, args: list[object]) -> list[object]`

Represents a symbolic `if` body.

What it means:

- Helion only emits `_if` when the branch condition is symbolic.
- If the condition is already concrete during tracing, Helion executes the chosen branch directly and no `_if` node is needed.

Parameters:

- `test: object`
  The branch condition, usually a symbolic or traced boolean.
- `graph_id: int`
  Index of the branch-body subgraph, usually an `IfGraphInfo`.
- `args: list[object]`
  Lifted values needed by that branch body.

Result:

- Returns the values written by that branch and later merged with `_phi`.

Notes:

- An `else` is represented by tracing the other branch separately, commonly using `_not(test)` and another `_if`-style subgraph construction internally.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- Lowered to `scf.if <cond> { ... }`.
- The current lowering only emits a side-effect-only `scf.if` with no result values.
- Branch placeholders are bound to outer SSA values before the inner `IfGraphInfo` is visited.
- No explicit `else` region is emitted by `visit_if`; the traced graphs already decide what body is represented by a given `_if`.

### `_phi(lhs: object, rhs: object) -> object`

Represents SSA merge at control-flow join points.

What it means:

- `_phi` says "the value after this control-flow join is either `lhs` or `rhs`, depending on which path executed".
- Helion uses it after loops and symbolic branches.

Parameters:

- `lhs: object`
  Value from one predecessor.
- `rhs: object`
  Value from the other predecessor.

Type constraints enforced by Helion:

- For `Tile` values: both sides must be tiles with the same `block_id`.
- For `torch.Tensor` values: same shape, dtype, and device.

Notes:

- `_phi` is marked side-effecting in the traced IR so it is not dropped too aggressively.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- No `helion.phi`-like MLIR op is emitted.
- The only supported lowering pattern is:
  `_phi(init, getitem(_for_loop(...), i))`
- In that case, the lowering just forwards the loop-result SSA because `scf.for iter_args` already encode the merge semantics.
- Any other `_phi` pattern is currently rejected.

### `_mask_to(tensor: torch.Tensor, other: float | bool) -> torch.Tensor`

Compiler-generated masking helper.

What it means:

- Helion inserts `_mask_to` when it needs out-of-bounds lanes to be replaced with a known fill value.
- This commonly appears around masked dot/reduction-style code.

Parameters:

- `tensor: torch.Tensor`
  Tensor whose masked-out lanes should be replaced.
- `other: float | bool`
  Replacement value for masked-out lanes.

Notes:

- This is not normally written by users directly.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- No masking MLIR is currently emitted.
- The lowering simply forwards the input tensor SSA unchanged.
- The file explicitly marks this as incomplete and leaves proper masking as future work.

## Memory and Tensor-View Ops

These are user-facing Helion APIs that appear explicitly in Device IR.

### `load(tensor, index, extra_mask=None, eviction_policy=None) -> torch.Tensor`

Defined in `helion.language.memory_ops`.

What it means:

- Load from a tensor using Helion indexing rules.
- Semantically this is the explicit API form of `tensor[index]` for kernel loads.

Parameters:

- `tensor: torch.Tensor | StackTensor`
  Source tensor-like object.
- `index: list[object]`
  Index list.
  Tiles are converted to size/index expressions by Helion before codegen.
  Tensor indices are allowed.
- `extra_mask: torch.Tensor | None = None`
  Optional additional boolean mask applied on top of Helion's automatic bounds masking.
- `eviction_policy: str | None = None`
  Optional Triton load eviction-policy hint.
  Helion recognizes short forms such as `"first"` and `"last"` and maps them to Triton-style names.

Notes:

Current lowering in `src/helion_mlir/ir_visitor.py`:

- Lowered to:
  `memref.subview` followed by `bufferization.to_tensor`.
- Scalar integer/scalar SSA indices are treated as rank-reducing indices with subview size `1`.
- Full slices use offset `0`, stride `1`, and use either a static dimension or a generated `memref.dim`.
- Tile/block indices are lowered as range accesses:
  offset = `%iv_block_<id> * block_size`
  size = `block_size`
- `tile.index + base` patterns are recognized specially as contiguous ranges.
- The result type is constructed as a tensor type matching the retained sliced dimensions.

### `store(tensor, index, value, extra_mask=None) -> None`

Defined in `helion.language.memory_ops`.

What it means:

- Store into a tensor using Helion indexing rules.
- Semantically this is the explicit API form of `tensor[index] = value`.

Parameters:

- `tensor: torch.Tensor | StackTensor`
  Destination tensor-like object.
- `index: list[object]`
  Index list.
- `value: torch.Tensor | torch.SymInt | float`
  Value to write.
  If `value` is a tensor with a different dtype, Helion casts it to the destination dtype first.
- `extra_mask: torch.Tensor | None = None`
  Optional additional mask for predicated stores.

Notes:

Current lowering in `src/helion_mlir/ir_visitor.py`:

- Lowered to:
  `memref.subview`
  `bufferization.to_buffer`
  `memref.copy`
- The destination subview indexing rules mirror `visit_load`.
- The source tensor value is converted to a buffer with the same subview memref type before the copy.

### `subscript(tensor: torch.Tensor, index: list[object]) -> torch.Tensor`

Defined in `helion.language.view_ops`.

What it means:

- Explicit kernel-tensor view indexing used for reshape-like indexing rather than memory load/store indexing.
- It is the IR form of expressions such as `x[:, None]`, `x[None, :]`, or similar pure view reshapes on already-loaded device tensors.

Parameters:

- `tensor: torch.Tensor`
  A kernel tensor, not a host argument tensor.
- `index: list[object]`
  Only `None` and `slice(None)` are supported here.
  This op is for adding/removing broadcast structure, not for general slicing.

Notes:

- In the examples, this appears in broadcast patterns such as `scale[:, None]`.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- First lowered through `tensor.extract_slice` for ordinary slicing and integer indexing.
- Then, if `None` is present, lowered through `tensor.expand_shape` to reinsert size-1 dimensions.
- Dynamic extents are resolved from Helion metadata when possible, otherwise with `tensor.dim`.

### `full(shape, value, dtype=torch.float32, device=None) -> torch.Tensor`

Defined in `helion.language.creation_ops`.

What it means:

- Create a device-local tensor filled with a constant value.
- This is used for temporary local tensors inside tiled regions.

Parameters:

- `shape: list[object]`
  Tensor shape.
  Tile objects are accepted at the Python surface and converted to sizes.
- `value: float`
  Fill value.
- `dtype: torch.dtype = torch.float32`
  Output element type.
- `device: torch.device | None = None`
  Optional device override.

Notes:

- `hl.zeros(shape, dtype, device)` is lowered to this op with `value = 0` or `0.0`.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- Rank-0 `full([] , value, ...)` lowers directly to `arith.constant`.
- Higher-rank `full(...)` lowers to:
  `tensor.empty`
  `arith.constant` for the fill value
  `linalg.fill`
- Dynamic dimensions are passed as operands to `tensor.empty`; static dimensions stay in the result type.

## Tile-Property Ops

These live in `helion.language.tile_ops` and are typically produced by tile property access.

### `tile_begin(tile) -> int`

What it means:

- Start offset of the current tile.
- This is the IR form of `tile.begin`.

Parameters:

- `tile: TileInterface`
  The tile whose start offset is requested.

Result:

- A symbolic integer representing the tile's starting index.

### `tile_id(tile) -> int`

What it means:

- Tile ordinal along that dimension.
- Equivalent to `tile.begin // tile.block_size`.
- This is the IR form of `tile.id`.

Parameters:

- `tile: TileInterface`
  The tile whose ordinal is requested.

Result:

- A symbolic integer tile id.

### `tile_index(tile) -> torch.Tensor`

What it means:

- Vector of element indices covered by the tile.
- This is the IR form of `tile.index`.

Parameters:

- `tile: TileInterface`
  The tile whose per-lane indices are requested.

Result:

- A 1-D tensor of indices spanning the tile.

Notes for all three tile ops:

- They are tied to Helion's block-id metadata.
- Helion uses origin metadata so these values can be reconstructed later during codegen.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- `tile_begin(tile)` lowers to `arith.muli %iv_block_<id>, <block_size_ssa> : index`.
- `tile_id(tile)` lowers directly to the loop IV `%iv_block_<id>`.
- `tile_index(tile)` does not lower to a real tensor-producing MLIR op by itself in this repo.
  It is recorded as a range-producing marker, initialized with a placeholder `arith.constant 0 : index`, and later consumed by the special `tile.index + base` pattern handling used by `load`, `store`, and `atomic_add`.

## Math / Reduction-Like Helion Ops

### `dot(mat1, mat2, acc=None, out_dtype=None) -> torch.Tensor`

Defined in `helion.language.matmul_ops`.

What it means:

- Helion's explicit matrix-multiply / batched-matrix-multiply op for device code.
- It is the IR form of `hl.dot(...)`.

Parameters:

- `mat1: torch.Tensor`
  2-D or 3-D tensor.
- `mat2: torch.Tensor`
  2-D or 3-D tensor.
  Must satisfy matrix multiplication shape compatibility.
- `acc: torch.Tensor | None = None`
  Optional accumulator tensor.
  If provided, result is accumulated into this tensor shape.
- `out_dtype: torch.dtype | None = None`
  Optional override for the output dtype used by the dot prior to accumulation.

Key constraints enforced by Helion:

- Supported input dtypes are restricted.
- `mat1.shape[-1]` must equal `mat2.shape[-2]`.
- If either input is batched, batch dimensions must be broadcast-compatible.
- If `acc` is provided, its rank, shape, and dtype must be valid for the dot.

Notes:

Current lowering in `src/helion_mlir/ir_visitor.py`:

- Lowered directly to `linalg.matmul`.
- The first two operands become `ins(...)`.
- The accumulator becomes the `outs(...)` operand.
- The result type is the accumulator tensor type.

### `atomic_add(target, index, value, sem="relaxed") -> torch.Tensor`

Defined in `helion.language.atomic_ops`.

What it means:

- Atomic read-modify-write add into a destination tensor.
- The return value is the old value at the destination.

Parameters:

- `target: torch.Tensor`
  Tensor being updated atomically.
- `index: list[object]`
  Indices selecting destination elements.
- `value: torch.Tensor | float`
  Value to add.
- `sem: str = "relaxed"`
  Memory ordering semantic.
  Allowed values are:
  `"relaxed"`, `"acquire"`, `"release"`, `"acq_rel"`.

Notes:

- In the current `split_k_matmul` example, this is used as the reduction writeback.

Current lowering in `src/helion_mlir/ir_visitor.py`:

- The destination is first lowered exactly like a tiled host-tensor access:
  `memref.subview`
- The update itself is then lowered as:
  `"loom.sum"(%subview, %value) : (<subview memref type>, <value tensor type>) -> ()`
- So the current backend does not emit a generic MLIR atomic op here; it emits the project-specific `loom.sum`.

## What Is Not Covered Here

The example graphs also contain non-ATen but non-Helion-specific FX targets such as:

- `operator.add`
- `operator.mul`
- `operator.floordiv`
- `operator.eq`
- `operator.getitem`

Those are generic Python / FX operations rather than Helion IR APIs, so they are better documented as ordinary scalar/index expressions instead of as Helion-specific IR nodes.
